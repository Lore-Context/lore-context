# Политика безопасности

Lore Context обрабатывает память, трассы, аудит-журналы и учётные данные интеграций. Рассматривайте
отчёты о безопасности как высокоприоритетные.

## Сообщение об уязвимости

Не открывайте публичные задачи для предполагаемых уязвимостей, утечек секретов, обходов аутентификации,
раскрытия данных или проблем с изоляцией тенантов.

Предпочтительный путь сообщения:

1. Используйте **GitHub приватное сообщение об уязвимости** для этого репозитория, если доступно.
2. Если приватная отчётность недоступна, свяжитесь с мейнтейнерами в частном порядке и укажите:
   - затронутую версию или коммит,
   - шаги воспроизведения,
   - ожидаемое воздействие,
   - задействованы ли настоящие секреты или персональные данные.

Мы стремимся подтвердить получение достоверных отчётов в течение 72 часов.

## Поддерживаемые версии

Lore Context в настоящее время является предрелизным альфа-ПО версии до 1.0. Исправления
безопасности сначала нацелены на ветку `main`. Помеченные релизы могут получать целевые патчи,
когда публичный релиз активно используется нижестоящими операторами.

| Версия | Поддерживается |
|---|---|
| v0.4.x-alpha | ✅ Активная |
| v0.3.x и более ранние | ❌ Только внутренний предрелиз |

## Встроенные средства защиты (v0.4.0-alpha)

Альфа-версия поставляется со следующими средствами эшелонированной защиты. Операторы должны
убедиться, что они активны в их развёртывании.

### Аутентификация

- **Bearer-токены API-ключей** (`Authorization: Bearer <key>` или заголовок `x-lore-api-key`).
- **Разделение ролей**: `reader` / `writer` / `admin`.
- **Ограничение по проекту**: записи в `LORE_API_KEYS` JSON могут включать список
  разрешений `projectIds: ["..."]`; мутации требуют совпадающего `project_id`.
- **Режим с пустыми ключами завершается ошибкой в продакшне**: при `NODE_ENV=production`
  и отсутствии настроенных ключей API отклоняет все запросы.
- **Обход loopback удалён**: предыдущие версии доверяли `Host: 127.0.0.1`; v0.4 использует
  только адрес сокета на уровне сокета.

### Ограничение частоты

- **Двойной бакет per-IP и per-key** с backoff при ошибках аутентификации.
- **По умолчанию**: 60 запросов/мин на IP для неаутентифицированных путей, 600 запросов/мин
  на аутентифицированный ключ.
- **5 ошибок аутентификации за 60 с → блокировка на 30 с** (возвращает 429).
- Настраивается: `LORE_RATE_LIMIT_PER_IP`, `LORE_RATE_LIMIT_PER_KEY`,
  `LORE_RATE_LIMIT_DISABLED=1` (только для разработки).

### Защита Dashboard

- **Промежуточное ПО HTTP Basic Auth** (`apps/dashboard/middleware.ts`).
- **Запуск в продакшне отклоняется** без
  `DASHBOARD_BASIC_AUTH_USER` и `DASHBOARD_BASIC_AUTH_PASS`.
- `LORE_DASHBOARD_DISABLE_AUTH=1` учитывается только вне продакшна.
- Серверный резервный вариант admin-ключа **удалён**: пользователь должен быть аутентифицирован
  через Basic Auth до того, как прокси dashboard внедряет upstream-учётные данные API.

### Защита контейнеров

- Все Dockerfile запускаются от непривилегированного пользователя `node`.
- `apps/api/Dockerfile` и `apps/dashboard/Dockerfile` объявляют `HEALTHCHECK`
  для `/health`.
- `apps/mcp-server` является только stdio — без сетевого слушателя — и не объявляет
  `HEALTHCHECK`.

### Управление секретами

- **Никаких жёстко закодированных учётных данных.** Все значения по умолчанию в
  `docker-compose.yml`, `docs/deployment/compose.private-demo.yml` и `.env.example`
  используют форму `${VAR:?must be set}` — запуск завершается ошибкой без явных значений.
- `scripts/check-env.mjs` отвергает значения-заглушки
  (`read-local`, `write-local`, `admin-local`, `change-me`, `demo`, `test`,
  `dev`, `password`) при `NODE_ENV=production`.
- Все документы развёртывания и примеры README очищены от буквальных демо-учётных данных.

### Управление

- **Сканирование тегов риска при каждой записи памяти**: обнаруживаются API-ключи, ключи AWS,
  JWT-токены, приватные ключи, пароли, электронные адреса, номера телефонов.
- **Шестисостояний конечный автомат** с явной таблицей допустимых переходов; недопустимые
  переходы вызывают исключение.
- **Эвристика отравления памяти**: доминирование одного источника + сопоставление паттернов
  императивных глаголов → флаг `suspicious`.
- **Неизменяемый аудит-журнал** добавляется при каждом переходе состояния.
- Высокорисковый контент автоматически направляется в `candidate` / `flagged` и удерживается
  от составления контекста до проверки.

### Защита MCP

- Каждый ввод MCP-инструмента **проверяется против zod-схемы** перед вызовом.
  Ошибки валидации возвращают JSON-RPC `-32602` с санированным списком проблем.
- **Все мутирующие инструменты** требуют строку `reason` не менее 8 символов и
  указывают `destructiveHint: true` в своей схеме.
- Ошибки upstream API **санируются** перед возвратом MCP-клиентам —
  сырой SQL, пути к файлам и трассировки стека очищаются.

### Логирование

- **Структурированный JSON-вывод** с корреляцией `requestId` по всей цепочке обработчиков.
- **Автоматическое редактирование** полей, соответствующих `content`, `query`, `memory`,
  `value`, `password`, `secret`, `token`, `key`. Фактическое содержимое записей памяти
  и запросов никогда не записывается в журналы.

### Границы данных

- Адаптер `agentmemory` зондирует upstream-версию при инициализации и предупреждает о
  несовместимости. `LORE_AGENTMEMORY_REQUIRED=0` переводит адаптер в тихий режим
  деградации, если upstream недоступен.
- Парсер тела запроса `apps/api` применяет ограничение `LORE_MAX_JSON_BYTES`
  (по умолчанию 1 МБ); слишком большие запросы возвращают 413.
- Пул соединений Postgres устанавливает `statement_timeout: 15000` для ограничения
  времени выполнения запросов.
- `LORE_REQUEST_TIMEOUT_MS` (по умолчанию 30 с) ограничивает каждый обработчик запроса;
  при превышении таймаута возвращается 504.

## Руководство по развёртыванию

- Не открывайте Lore публично без настроенных `LORE_API_KEYS`.
- Предпочтительно использовать ключи `reader` / `writer` / `admin` с разделением ролей.
- **Всегда устанавливайте** `DASHBOARD_BASIC_AUTH_USER` и `DASHBOARD_BASIC_AUTH_PASS`
  в продакшне.
- **Генерируйте ключи с помощью `openssl rand -hex 32`**. Никогда не используйте
  значения-заглушки из примеров.
- Держите конечные точки raw `agentmemory` закрытыми; обращайтесь к ним только через Lore.
- Держите маршруты dashboard, governance, import/export, sync и аудита за уровнем сетевого
  контроля доступа (Cloudflare Access, AWS ALB, Tailscale ACL или аналогичным) для любого
  воздействия не через loopback.
- **Запустите `node scripts/check-env.mjs` перед запуском API в продакшне.**
- **Никогда не коммитируйте** продакшн-файлы `.env`, API-ключи провайдеров, облачные
  учётные данные, данные eval с клиентским содержимым или приватные экспорты памяти.

## Временная шкала раскрытия

Для подтверждённых высокоimpact уязвимостей:

- 0 дней: отчёт подтверждён.
- 7 дней: классификация по триажу и серьёзности передаётся репортёру.
- 30 дней: координированное публичное раскрытие (или расширено по взаимному соглашению).
- 30+ дней: выпуск CVE для уязвимостей среднего+ уровня серьёзности, если применимо.

Для менее серьёзных проблем ожидайте решения в следующем минорном релизе.
