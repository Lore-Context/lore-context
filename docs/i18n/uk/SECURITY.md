# Політика безпеки

Lore Context обробляє пам'ять, траси, журнали аудиту та облікові дані інтеграції.
Ставтеся до звітів безпеки як до високопріоритетних.

## Повідомлення про вразливість

Не відкривайте публічний тікет для підозрюваних вразливостей, витоків секретів,
обходів автентифікації, розкриття даних або проблем ізоляції орендарів.

Рекомендований шлях звітування:

1. Використовуйте **приватне звітування про вразливості GitHub** для цього репозиторію,
   якщо доступно.
2. Якщо приватне звітування недоступне, зверніться до супровідників приватно та вкажіть:
   - версію або коміт, що зачеплені,
   - кроки відтворення,
   - очікуваний вплив,
   - чи залучені реальні секрети або персональні дані.

Ми прагнемо підтвердити достовірні звіти протягом 72 годин.

## Підтримувані версії

Lore Context наразі є альфа-програмним забезпеченням до версії 1.0. Виправлення
безпеки в першу чергу орієнтовані на гілку `main`. Теговані випуски можуть отримувати
цільові патчі, коли публічний випуск активно використовується downstream операторами.

| Версія | Підтримується |
|---|---|
| v0.4.x-alpha | ✅ Активна |
| v0.3.x та раніше | ❌ Лише внутрішній передвипуск |

## Вбудоване зміцнення (v0.4.0-alpha)

Альфа постачається з такими засобами поглибленого захисту. Оператори повинні
перевірити, що вони активні в їхньому розгортанні.

### Автентифікація

- **Токени-носії API-ключів** (`Authorization: Bearer <key>` або заголовок
  `x-lore-api-key`).
- **Розподіл ролей**: `reader` / `writer` / `admin`.
- **Обмеження до проекту**: записи `LORE_API_KEYS` JSON можуть містити список дозволів
  `projectIds: ["..."]`; мутації вимагають відповідного `project_id`.
- **Режим порожніх ключів блокується у виробництві**: при `NODE_ENV=production` без
  налаштованих ключів API відхиляє всі запити.
- **Обхід зворотного зв'язку видалено**: попередні версії довіряли
  `Host: 127.0.0.1`; v0.4 використовує лише адресу на рівні сокета.

### Обмеження частоти

- **Подвійний кошик per-IP та per-key** з відступом при помилці автентифікації.
- **За замовчуванням**: 60 запитів/хвилину на IP для неавтентифікованих шляхів, 600
  запитів/хвилину на автентифікований ключ.
- **5 невдалих спроб автентифікації протягом 60с → блокування на 30с** (повертає 429).
- Налаштовується: `LORE_RATE_LIMIT_PER_IP`, `LORE_RATE_LIMIT_PER_KEY`,
  `LORE_RATE_LIMIT_DISABLED=1` (лише для розробки).

### Захист дашборду

- **Проміжне ПЗ HTTP Basic Auth** (`apps/dashboard/middleware.ts`).
- **Запуск у виробництві відмовляє** без `DASHBOARD_BASIC_AUTH_USER` та
  `DASHBOARD_BASIC_AUTH_PASS`.
- `LORE_DASHBOARD_DISABLE_AUTH=1` враховується лише поза виробництвом.
- Резервний адміністративний ключ на стороні сервера **видалено**: користувач повинен
  бути автентифікований через Basic Auth перед тим, як проксі дашборду вставляє
  upstream API облікові дані.

### Зміцнення контейнерів

- Всі Dockerfiles запускаються від непривілейованого користувача `node`.
- `apps/api/Dockerfile` та `apps/dashboard/Dockerfile` оголошують `HEALTHCHECK`
  для `/health`.
- `apps/mcp-server` є лише stdio — без мережевого прослуховувача — і не оголошує
  `HEALTHCHECK`.

### Управління секретами

- **Нуль жорстко закодованих облікових даних.** Всі значення за замовчуванням у
  `docker-compose.yml`, `docs/deployment/compose.private-demo.yml` та `.env.example`
  використовують форму `${VAR:?must be set}` — запуск завершується невдачею без явних
  значень.
- `scripts/check-env.mjs` відхиляє значення заповнювачів
  (`read-local`, `write-local`, `admin-local`, `change-me`, `demo`, `test`,
  `dev`, `password`) при `NODE_ENV=production`.
- Всі документи розгортання та приклади README очищено від буквальних демо облікових
  даних.

### Управління

- **Сканування тегів ризику при кожному записі пам'яті**: виявляються API ключі, ключі
  AWS, JWT токени, приватні ключі, паролі, електронні пошти, номери телефонів.
- **Шестистанний автомат** з явною таблицею допустимих переходів; недопустимі переходи
  спричиняють виключення.
- **Евристика отруєння пам'яті**: домінування одного джерела + зіставлення шаблонів
  наказових дієслів → мітка `suspicious`.
- **Незмінний журнал аудиту** додається при кожному переході стану.
- Контент з підвищеним ризиком автоматично скеровується до `candidate` / `flagged` та
  затримується від компоновки контексту до перегляду.

### Зміцнення MCP

- Кожне вхідне значення інструменту MCP **перевіряється по схемі zod** перед
  викликом. Невдалі перевірки повертають JSON-RPC `-32602` з очищеним списком проблем.
- **Всі мутаційні інструменти** вимагають рядок `reason` щонайменше 8 символів та
  відображають `destructiveHint: true` у своїй схемі.
- Помилки upstream API **очищуються** перед поверненням клієнтам MCP — необроблений
  SQL, шляхи до файлів та трасування стека видаляються.

### Логування

- **Структурований JSON вивід** з кореляцією `requestId` по всьому ланцюгу обробника.
- **Автоматична редакція** полів, що відповідають `content`, `query`, `memory`,
  `value`, `password`, `secret`, `token`, `key`. Фактичний вміст записів пам'яті та
  запитів ніколи не записується в журнали.

### Межі даних

- Адаптер `agentmemory` перевіряє upstream версію при ініціалізації та попереджає про
  несумісність. `LORE_AGENTMEMORY_REQUIRED=0` перемикає адаптер на тихий режим
  деградації, якщо upstream недосяжний.
- Парсер тіла запиту `apps/api` забезпечує ліміт `LORE_MAX_JSON_BYTES` (за
  замовчуванням 1 МіБ); надмірно великі запити повертають 413.
- Пул підключень Postgres встановлює `statement_timeout: 15000` для обмеження часу
  запиту.
- `LORE_REQUEST_TIMEOUT_MS` (за замовчуванням 30с) обмежує кожен обробник запиту;
  тайм-аути повертають 504.

## Рекомендації з розгортання

- Не відкривайте Lore зовні без налаштованих `LORE_API_KEYS`.
- Віддавайте перевагу **розподіленим ролям** ключів `reader` / `writer` / `admin`.
- **Завжди встановлюйте** `DASHBOARD_BASIC_AUTH_USER` та `DASHBOARD_BASIC_AUTH_PASS`
  у виробництві.
- **Генеруйте ключі за допомогою `openssl rand -hex 32`**. Ніколи не використовуйте
  значення заповнювачів з прикладів.
- Тримайте raw `agentmemory` ендпоінти приватними; отримуйте до них доступ лише через
  Lore.
- Тримайте маршрути дашборду, управління, імпорту/експорту, синхронізації та аудиту
  за шаром контролю мережевого доступу (Cloudflare Access, AWS ALB, Tailscale ACL
  або подібне) для будь-якого не-loopback відкриття.
- **Виконуйте `node scripts/check-env.mjs` перед запуском API у виробництві.**
- **Ніколи не комітьте** виробничі файли `.env`, API ключі провайдерів, хмарні
  облікові дані, дані eval з клієнтським контентом або приватні експорти пам'яті.

## Терміни розкриття

Для підтверджених вразливостей з великим впливом:

- 0 днів: звіт підтверджено.
- 7 днів: тріаж та класифікація серйозності повідомляються репортеру.
- 30 днів: координоване публічне розкриття (або продовжено за взаємною згодою).
- 30+ днів: присвоєння CVE для середньої та вищої серйозності, якщо застосовується.

Для менш серйозних проблем очікуйте вирішення в наступному мінорному випуску.
