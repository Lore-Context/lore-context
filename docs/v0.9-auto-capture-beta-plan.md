# Lore v0.9 Auto-Capture Beta Plan

Last updated: 2026-05-01

Status: planned sprint, not shipped

Owner: Lore project lead

Primary private repo: `Lore-Context/lore-cloud`

Public repo role: public trust surface, open-core compatibility, docs boundary

## 0. Executive Decision

`v0.8.0-beta` proved that the closed-source personal cloud runtime can be
deployed, protected, and verified on the AWS Singapore production stack. `v0.9`
must prove a different product claim:

**A non-technical user can connect Lore once, then have useful memories captured
automatically from the AI tools they already use, while retaining visible
control over what is stored and reused.**

Version name:

**Lore v0.9 Auto-Capture Beta**

Product slogan:

**All your agents. One shared memory.**

Product principle:

**Capture first, control always, recall everywhere.**

The v0.9 decision is to prioritize auto-capture, hosted MCP, browser capture,
and one production-grade document connector. Team workspaces, full BYOC,
advanced data protection, and broad public GA are intentionally deferred until
the private beta has real retention, cost, and support evidence.

## 2. Product Strategy

### 2.1 Target User

Primary user:

- non-technical or lightly technical AI user;
- uses Claude Code, Codex, Cursor, Qwen/OpenCode, ChatGPT, Claude.ai, Gemini,
  or Perplexity;
- wants AI tools to remember preferences, projects, decisions, tasks, and
  context without manually maintaining markdown files or MCP configs;
- cares about privacy/control once they realize memory is being stored.

Secondary user:

- technical early adopter who wants cross-agent memory and auditability;
- design partner who can tolerate beta rough edges and provide activation data.

Not the v0.9 primary user:

- enterprise buyer requiring SAML, SOC 2 procurement, HIPAA BAA, BYOC, custom
  cloud, or legal review before first use.

### 2.2 v0.9 Product Promise

User-facing promise:

1. Connect Lore once.
2. Keep using your normal AI tools.
3. Lore quietly captures useful context.
4. You approve, edit, delete, pause, or export anything.
5. Every connected agent can recall the same memory.
6. Evidence Ledger shows why a memory was used.

### 2.3 Success Metrics

Activation:

- user to first connected agent: `<5 minutes`;
- user to first captured memory: `<15 minutes`;
- user to first cross-agent recall: `<30 minutes`;
- first 50 beta users: `>=75%` connect at least one source without maintainer
  screen-share;
- first 50 beta users: `>=60%` produce at least one approved memory within 24
  hours.

Retention:

- day-2 return: `>=35%`;
- day-7 return: `>=20%`;
- at least `25%` of active beta users connect two or more capture sources.

Trust/control:

- `100%` of auto-captured items have source refs, timestamps, source type, and
  delete/export path;
- `100%` of recall responses produce Evidence Ledger traces;
- privacy pause/delete/export flows pass production smoke tests before beta
  expansion.

Cost:

- free-tier user cost target: `<$0.20/user/month` blended infra and model cost
  during beta;
- no surprise overage billing in v0.9;
- caps must throttle gracefully before runaway ingestion or recall.

### 4.2 Capture Source Model

Every captured item must be attached to a `capture_source`.

Required source fields:

- `id`;
- `vault_id`;
- `source_type`: `agent_session`, `browser_chat`, `connector_document`,
  `hosted_mcp`, `manual_api`;
- `provider`: `claude_code`, `codex`, `cursor`, `qwen`, `chatgpt_web`,
  `claude_web`, `gemini_web`, `google_drive`, `notion`, etc.;
- `display_name`;
- `status`: `active`, `paused`, `private_mode`, `revoked`, `error`;
- `permissions`: JSON permission envelope;
- `raw_archive_policy`: `none`, `metadata_only`, `summary_only`,
  `encrypted_raw`, `plain_raw_for_beta_debug`;
- `created_at`, `last_seen_at`, `paused_at`, `revoked_at`.

Design rule:

`paused` and `private_mode` must be enforced at ingestion and at UI level. They
cannot be dashboard-only states.

### 4.3 Capture Event Pipeline

New canonical event shape:

```json
{
  "source_id": "src_...",
  "external_event_id": "claude-session-path-and-offset",
  "occurred_at": "2026-05-01T00:00:00Z",
  "event_type": "session_delta",
  "actor": "user|assistant|tool|system",
  "content_ref": {
    "kind": "inline|blob|local_ref",
    "sha256": "..."
  },
  "redaction_state": "redacted|raw_allowed|metadata_only",
  "idempotency_key": "..."
}
```

Pipeline stages:

1. Ingest event.
2. Validate source status and permissions.
3. Redact obvious secrets and private content.
4. Deduplicate by source and external event id.
5. Normalize into `capture_events`.
6. Group into candidate memory batches.
7. Extract candidate facts, preferences, decisions, tasks, and project context.
8. Write candidates to Memory Inbox.
9. User or policy approves candidates into durable memory/profile.
10. Recall uses approved memory with Evidence Ledger trace.

### 4.4 Universal Session Watcher

`lore watch` becomes the primary local daemon.

P0 watched sources:

- Claude Code session directories;
- Codex session directories;
- Cursor agent/chat storage where available and stable;
- Qwen/OpenCode session logs where available.

Implementation details:

- detect installed agents at startup;
- register each agent as a `capture_source`;
- maintain per-source checkpoint file locally;
- upload deltas, not whole files;
- batch uploads to reduce API calls;
- backoff on network failure;
- never upload during `private_mode`;
- keep local redacted logs for support only when user enables diagnostics;
- report `connected`, `watching`, `last_upload`, `last_error`,
  `captured_today`, and `pending_review` in `lore status`.

Local state:

- macOS Keychain for cloud credentials;
- `~/.lore/sources.json` for non-secret source metadata;
- `~/.lore/checkpoints/*.json` for offsets and hashes;
- config backups under `~/.lore/backups/`.

### 4.5 Hosted MCP

v0.8 keeps local stdio bridge. v0.9 adds hosted MCP for ordinary users.

Endpoint shape:

- `https://mcp.lorecontext.com/mcp`
- OAuth-based auth by default;
- API-key or install-token auth for CLI automation;
- project/vault scoping through explicit headers or OAuth claims;
- user-visible connected-client list in dashboard.

MCP tools:

- `memory.add_candidate`;
- `memory.recall`;
- `memory.inbox_list`;
- `memory.inbox_approve`;
- `memory.inbox_reject`;
- `memory.delete`;
- `source.pause`;
- `source.resume`;
- `evidence.trace_get`;
- `profile.get`;
- `profile.update_candidate`.

Protocol requirements:

- implement OAuth protected resource metadata;
- return correct `WWW-Authenticate` metadata on `401`;
- support authorization server metadata;
- support dynamic client registration only if it can be implemented safely;
- otherwise provide client-specific one-click config in dashboard.

### 4.6 Browser Extension

Chrome extension MVP:

- Manifest V3;
- background service worker;
- content scripts for allowlisted AI sites;
- popup with current-site capture status;
- options page for connected vault, pause, private mode, and diagnostics;
- side panel only if it materially improves review workflow.

P0 supported sites:

- `chatgpt.com` / `chat.openai.com`;
- `claude.ai`;
- `gemini.google.com`;
- `perplexity.ai`.

Technical constraints:

- MV3 service workers are not persistent; do not rely on globals for state;
- register listeners synchronously;
- store state in `chrome.storage.local`;
- use alarms for retry/sync timers;
- use explicit host permissions and optional host permissions where possible;
- content scripts extract conversation DOM into a normalized local envelope;
- background worker sends redacted capture events to Lore Cloud;
- never capture password fields, hidden inputs, unrelated page content, or
  non-allowlisted domains.

UX requirements:

- capture indicator visible but quiet;
- one-click pause;
- private mode for current tab/domain;
- "Review in Lore" link;
- no default capture before user authorizes the source.

### 4.7 Connector Framework

P0 is the framework plus one excellent connector.

Recommended first connector:

1. Google Drive if OAuth credentials and verification are ready.
2. Notion if Google OAuth verification blocks deployment.

Connector interface:

```ts
interface ConnectorProvider {
  provider: string;
  createAuthorizationUrl(input): Promise<AuthUrl>;
  handleCallback(input): Promise<Connection>;
  backfill(connectionId): Promise<SyncResult>;
  handleWebhook(input): Promise<WebhookResult>;
  syncIncremental(connectionId): Promise<SyncResult>;
  revoke(connectionId): Promise<void>;
}
```

Required connector tables:

- `connector_connections`;
- `connector_oauth_tokens`;
- `connector_sync_jobs`;
- `connector_sync_checkpoints`;
- `connector_documents`;
- `connector_webhook_events`;
- `connector_errors`.

Google Drive notes:

- use change notifications where feasible;
- maintain start page token / checkpoint;
- ingest Docs, Sheets, Slides metadata first, then content extraction;
- respect shared drive boundaries;
- allow folder-level scoping to cap cost.

Notion notes:

- verify webhook subscription;
- require explicit page/database access;
- support comments only if capability is enabled;
- store block tree checksum;
- support manual resync.

### 4.8 Memory Inbox 2.0

Memory Inbox is the control surface for auto-capture.

Candidate states:

- `pending`;
- `auto_approved`;
- `approved`;
- `edited`;
- `ignored`;
- `deleted`;
- `conflict`;
- `sensitive_review`;
- `stale_review`.

Candidate types:

- user preference;
- project fact;
- decision;
- task/follow-up;
- relationship;
- credential-like sensitive candidate;
- contradiction;
- stale correction;
- document summary;
- web-chat summary.

User actions:

- approve;
- edit and approve;
- reject;
- delete source item;
- pause source;
- mark private;
- export;
- report bad memory.

Policy:

- safe low-risk candidates can be auto-approved only after the user enables
  the toggle;
- sensitive candidates never auto-approve;
- rejected candidates should train per-user filtering rules;
- every approval/rejection writes an audit event.

### 4.9 Recall and Evidence Ledger

v0.9 recall must be source-aware and confidence-aware.

Retrieval layers:

1. profile facts;
2. recent agent sessions;
3. project memories;
4. connector documents;
5. browser conversation summaries;
6. explicit user-pinned memories.

Ranking signals:

- semantic similarity;
- keyword match;
- recency;
- source trust;
- user approval state;
- conflict/stale flags;
- current agent/project scope;
- prior used/ignored feedback.

Evidence Ledger must record:

- query;
- requesting agent/client;
- vault and project scope;
- candidate memories;
- retrieved memories;
- used memories;
- ignored memories;
- stale/conflict/sensitive flags;
- source refs and timestamps;
- user-visible explanation.

Stale policy:

- code/project memories require citation/source validation when possible;
- low-confidence memories decay over time;
- old memories can move to `stale_review` instead of being silently used;
- user-approved pinned memories do not auto-expire.

### 4.10 Pricing and Usage Guardrails

Pricing should be aggressive but capped.

Proposed beta packaging:

| Plan | Price | Limits | Purpose |
|---|---:|---|---|
| Free | `$0` | 1M ingest tokens/month, 10K recalls/month, 3 agents, browser extension, 1 connector preview, 30-day Evidence Ledger | Maximize adoption and data gravity |
| Personal | `$3/mo` | 10M ingest, 100K recalls, all personal agents, 2 connectors, 1-year Evidence Ledger | Clear low-price differentiation |
| Pro | `$9/mo` | 50M ingest, 500K recalls, 5 connectors, auto export backup, long retention | Power users and serious builders |
| Team waitlist | not launched | shared vaults, admin, audit export | Validate demand before building |

Rules:

- no surprise overage billing in v0.9;
- throttle first, upsell second;
- operator sees cost/user/source;
- dashboard shows usage before limit;
- raw archive storage must be disabled or tightly capped by default.

## 6. Security and Privacy Requirements

Minimum bar:

- explicit user authorization per source;
- per-source pause and revoke;
- delete memory, delete source, delete vault;
- MIF export;
- token redaction in logs;
- OAuth tokens encrypted at rest;
- browser extension stores only minimal local state;
- connector scopes are minimal;
- raw archive disabled by default;
- sensitive candidate detector before Memory Inbox;
- audit events for capture, approval, deletion, export, token creation, token
  revocation, and connector authorization.

Production boundary:

- Cloudflare Access can remain on dashboard during private beta;
- public self-serve signup stays invite-gated;
- no public marketing claim that Lore is generally available SaaS.

## 10. Risks

| Risk | Mitigation |
|---|---|
| Auto-capture feels creepy | explicit onboarding, visible source list, pause/private mode, Memory Inbox before durable approval |
| Browser DOM extraction breaks often | support only four sites first; use versioned extractors and graceful fallback |
| Connector OAuth verification blocks release | make Google Drive/Notion swappable; ship framework plus whichever connector is credential-ready |
| Ingestion cost grows faster than free-tier economics | default summaries over raw archive, caps, batching, per-source limits, operator cost view |
| Hosted MCP auth compatibility varies by client | keep local stdio bridge; implement spec metadata; document client-specific fallback |
| Recall quality is noisy | review-before-approve, source-aware ranking, stale/conflict flags, user feedback loop |
| Public/private claims blur | keep v0.9 in closed-source beta docs; public site can say private beta only |

## 11. Release Decision

Ship v0.9 only if all are true:

- at least one auto-capture path works end-to-end without maintainer
  screen-share;
- hosted MCP or local bridge connects at least two agents to one vault;
- Memory Inbox can approve/edit/reject/delete production candidates;
- Evidence Ledger records cross-agent recall;
- one connector or browser extension capture path is production-verified;
- usage caps and privacy controls are production-verified;
- no secrets are present in docs, logs committed to repo, or release artifacts.

If these are not true by the end of Week 8, do not rebrand as GA. Cut a
`v0.9.0-beta` private release with honest known gaps and continue beta
validation.
